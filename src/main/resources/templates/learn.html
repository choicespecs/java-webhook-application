Understanding Webhooks

A practical guide using this application as a live example.

What is a Webhook?

A webhook is a way for one system to notify another system that something happened — in real time, automatically, without polling.

Polling (old way)

Your app repeatedly asks: "Did anything happen yet?" — every few seconds, forever. Wasteful and slow.

Webhooks (modern way)

The source system calls you the moment something happens. One HTTP POST, instant notification.

Real-world examples
  • Stripe sends a webhook when a payment succeeds or fails
  • GitHub sends a webhook when code is pushed or a PR is opened
  • Shopify sends a webhook when a new order is placed
  • Twilio sends a webhook when an SMS is received

How Webhooks Work

A webhook is simply an HTTP POST request. Here is the end-to-end flow:

1. An event occurs in the source system
A user places an order, a payment is charged, a file is uploaded, etc.
Trigger
2. Source sends an HTTP POST to your endpoint
The source system makes a POST request with a JSON body describing the event, and optionally a signature header for security.
POST https://your-app.com/api/webhook/receive
Content-Type: application/json
X-Webhook-Signature: sha256=abc123...
HTTP POST
3. Your receiver validates & acknowledges immediately
You verify the signature, return HTTP 202 Accepted within a few seconds, then process asynchronously. The source expects a fast response — if you take too long, it will retry.
HTTP 202
4. Your system processes the event asynchronously
Update a database, send an email, trigger another service. Processing happens in the background — decoupled from the acknowledgment.
Async

How This Application Works

Here is exactly what happens when you send a request to POST /api/webhook/receive:

# Component What it does
1 WebhookController Receives the HTTP POST, reads the X-Webhook-Signature header, validates the request body with Spring Validation (@Valid)
2 WebhookService Verifies the HMAC-SHA256 signature if provided, creates a WebhookRecord with status QUEUED
3 WebhookHistoryService Stores the record in an in-memory ConcurrentLinkedDeque (up to 1,000 records)
4 WebhookQueue Places the record on a LinkedBlockingQueue (capacity 500). Three background threads pick it up for processing
5 Worker threads Each thread calls WebhookService.processRecord() — retrying up to 3 times with exponential backoff on failure
6 WebSocket broadcast After each state change, updated stats and history are pushed to all connected browsers over STOMP/SockJS — no page refresh needed
Why return 202 before processing? The source system (e.g. Stripe) has a short timeout (often 5–30 seconds). If you do heavy work synchronously — like sending emails or hitting a slow database — you risk timing out, causing the source to think delivery failed and retry. By queuing immediately and returning 202, you decouple receipt from processing.

Webhook Lifecycle

Every webhook record in this app passes through these states:

QUEUED
Received & waiting
PROCESSING
Worker picked up
SUCCESS
Processed OK
FAILED
Max retries exceeded

The History page shows all records with their current status. A badge color tells you at a glance where each webhook stands.

Security & Signatures

Without a signature, any client on the internet can POST to your webhook endpoint and pretend to be GitHub, Stripe, etc. HMAC-SHA256 signatures let you verify that a request genuinely came from the trusted source.

How HMAC-SHA256 works
Sender (source system)
  1. Serializes the payload to JSON
  2. Computes HMAC-SHA256(json, sharedSecret)
  3. Sends signature in header: X-Webhook-Signature: sha256=<hex>
Receiver (this app)
  1. Reads the raw request body
  2. Computes the same HMAC-SHA256 using the shared secret
  3. Compares — if they match, the payload is authentic
// WebhookService.java — signature generation
Mac mac = Mac.getInstance("HmacSHA256");
mac.init(new SecretKeySpec(secret.getBytes(), "HmacSHA256"));
byte[] hash = mac.doFinal(json.getBytes(StandardCharsets.UTF_8));
String signature = "sha256=" + HexFormat.of().formatHex(hash);
Important: The signature is optional in this app — validation only runs when the X-Webhook-Signature header is present. In production, always require and verify signatures. The shared secret is configured via webhook.secret in application.properties.

Retry Logic & Reliability

Networks fail. Downstream services crash. A robust webhook receiver should handle transient errors gracefully without overwhelming the upstream system. This app uses exponential backoff with a retry cap.

Retry schedule
Attempt 1
Immediate
Retry 1
Wait 1 second
Retry 2
Wait 2 seconds
Retry 3
Wait 4 seconds → FAILED
Why exponential backoff?

If many webhooks fail at once and all retry immediately, you create a thundering herd — hammering an already-struggling service. Spreading retries out with increasing delays gives the downstream service time to recover.

Why a retry cap?

Retrying indefinitely is rarely helpful. After 3 attempts the webhook is marked FAILED and stored in history so you can inspect it, without burning threads forever on a permanently broken event.

// WebhookQueue.java — retry loop
for (int attempt = 0; attempt <= MAX_RETRIES; attempt++) {
  if (attempt > 0) Thread.sleep(BACKOFF_MS[attempt - 1]); // 1000, 2000, 4000 ms
  webhookService.processRecord(record); // throws on failure
  return; // success — stop retrying
}

Real-time Updates via WebSockets

The dashboard updates instantly when a new webhook arrives — without polling. This is done with WebSockets using the STOMP protocol over SockJS.

1. Browser connects
Opens a persistent connection to /ws via SockJS
2. Browser subscribes
Subscribes to /topic/stats and /topic/history
3. Server pushes updates
After each webhook, the server broadcasts to all subscribers
// dashboard.js — STOMP subscription
var socket = new SockJS('/ws');
stompClient = Stomp.over(socket);
stompClient.connect({}, function(frame) {
  stompClient.subscribe('/topic/stats', function(msg) { updateStats(JSON.parse(msg.body)); });
  stompClient.subscribe('/topic/history', function(msg) { updateTable(JSON.parse(msg.body)); });
});
SockJS fallback: SockJS automatically falls back to long-polling if WebSockets are not available (e.g. behind certain proxies). The green/red Connected badge in the dashboard navbar shows the live connection status.
